# 보안 설계 문서

**대규모 트래픽 티켓 구매 웹 서비스**

---

## 1. 문서 개요

### 1.1 목적

본 문서는 Next.js(App Router) 기반 티켓 구매 웹 서비스의 보안 아키텍처를 정의한다. 대규모 트래픽 환경에서 발생 가능한 주요 보안 위협을 식별하고, 각 위협에 대한 구체적인 방어 전략을 제시한다.

### 1.2 보안 목표

- 사용자 인증 정보의 안전한 저장 및 전송
- 세션 하이재킹 및 CSRF 공격 방어
- XSS 공격을 통한 데이터 유출 방지
- SSR 환경에서의 민감 데이터 보호
- 대기열 토큰 위조 및 재사용 방지

### 1.3 기술 스택

| 영역 | 기술 |
|------|------|
| Frontend | Next.js (App Router), TypeScript |
| Backend | Express.js |
| Auth | JWT + HttpOnly Cookie |
| Infra | CloudFront / Cloudflare Workers, Nginx |

## 2. 인증 및 세션 관리

### 2.1 JWT 기반 인증 전략

#### 2.1.1 JWT 구조

JWT는 Access Token과 Refresh Token으로 구성되며, 각각 다른 목적과 유효 기간을 가진다.

| 토큰 타입 | 유효 기간 | 저장 위치 | 용도 |
|-----------|-----------|-----------|------|
| Access Token | 15분 | HttpOnly Cookie | API 인증 |
| Refresh Token | 7일 | HttpOnly Cookie | 토큰 갱신 |

#### 2.1.2 JWT Payload 구조

Access Token의 Payload는 다음 정보를 포함한다.

- `userId`: 사용자 고유 식별자
- `email`: 사용자 이메일
- `role`: 사용자 권한 (user, admin)
- `iat`: 토큰 발급 시간 (Issued At)
- `exp`: 토큰 만료 시간 (Expiration)

#### 2.1.3 토큰 갱신 메커니즘

Access Token 만료 시 Refresh Token을 이용해 자동으로 새로운 Access Token을 발급받는다. 이 과정은 사용자가 인지하지 못하도록 백그라운드에서 처리된다.

### 2.2 HttpOnly Cookie 전략

#### 2.2.1 HttpOnly Cookie 사용 이유

JWT를 localStorage나 sessionStorage에 저장하지 않고 HttpOnly Cookie에 저장하는 이유는 다음과 같다.

- **XSS 공격 방어**: JavaScript로 쿠키에 접근할 수 없어 토큰 탈취 불가
- **자동 전송**: 브라우저가 자동으로 쿠키를 요청에 포함
- **Secure 플래그 사용 가능**: HTTPS 환경에서만 전송

#### 2.2.2 Cookie 설정

모든 인증 쿠키는 다음 속성을 반드시 포함한다.

| 속성 | 설명 |
|------|------|
| HttpOnly | JavaScript로 쿠키 접근 차단 |
| Secure | HTTPS 환경에서만 쿠키 전송 |
| SameSite | Strict 또는 Lax - CSRF 공격 방어 |
| Path | 쿠키가 전송될 경로 지정 (보통 /) |

## 3. CSRF (Cross-Site Request Forgery) 방어

### 3.1 CSRF 공격 시나리오

공격자가 악의적인 웹사이트를 만들어 사용자가 로그인된 상태에서 해당 사이트를 방문하도록 유도한다. 이때 공격자의 사이트에서 사용자의 브라우저를 통해 정상 서비스로 요청을 보내면, 브라우저는 자동으로 쿠키를 포함하여 요청을 전송한다. 이를 통해 공격자는 사용자 권한으로 티켓 구매 등의 작업을 수행할 수 있다.

### 3.2 방어 전략

#### 3.2.1 SameSite Cookie 속성

쿠키에 `SameSite=Strict` 또는 `SameSite=Lax`를 설정하여 Cross-Origin 요청에서 쿠키가 자동으로 전송되지 않도록 한다.

- **SameSite=Strict**: 모든 Cross-Origin 요청에서 쿠키 전송 차단
- **SameSite=Lax**: GET 요청에만 쿠키 전송 허용 (권장)

#### 3.2.2 CSRF Token 사용

중요한 상태 변경 작업(티켓 구매, 결제 등)에는 추가로 CSRF Token을 사용한다. 서버에서 생성한 CSRF Token을 폼에 포함시키고, 요청 시 토큰을 함께 전송하여 검증한다.

- Server Component에서 CSRF Token 생성
- Hidden Input 또는 Custom Header로 전송
- 서버에서 토큰 유효성 검증

#### 3.2.3 Origin/Referer 검증

API 서버에서 요청의 Origin 또는 Referer 헤더를 검증하여 허용된 도메인에서만 요청이 오도록 제한한다.

## 4. XSS (Cross-Site Scripting) 방어

### 4.1 XSS 공격 유형

#### 4.1.1 Reflected XSS

사용자의 입력이 즉시 응답에 반영되는 경우 발생한다. 예를 들어 검색어를 URL 파라미터로 받아 그대로 화면에 출력할 때, 악의적인 스크립트가 실행될 수 있다.

#### 4.1.2 Stored XSS

공격자가 악의적인 스크립트를 데이터베이스에 저장하고, 다른 사용자가 해당 데이터를 조회할 때 스크립트가 실행되는 방식이다. 티켓 후기나 댓글 기능에서 발생할 수 있다.

#### 4.1.3 DOM-based XSS

클라이언트 측 JavaScript에서 사용자 입력을 직접 DOM에 삽입할 때 발생한다. `innerHTML`을 사용하여 HTML을 직접 렌더링하는 경우 위험하다.

### 4.2 방어 전략

#### 4.2.1 입력 검증 및 Sanitization

- 모든 사용자 입력에 대해 서버 측에서 검증 수행
- HTML 태그 및 특수 문자 이스케이프 처리
- DOMPurify 등의 라이브러리를 사용하여 HTML Sanitization

#### 4.2.2 React의 기본 보호

React는 기본적으로 모든 값을 렌더링할 때 자동으로 이스케이프 처리한다. 그러나 다음 상황에서는 주의가 필요하다.

- `dangerouslySetInnerHTML` 사용 금지 (불가피한 경우 Sanitize 필수)
- `href`에 `javascript:` 스키마 입력 차단
- 사용자 입력을 `eval()` 또는 `Function()` 생성자에 전달 금지

#### 4.2.3 Content Security Policy (CSP)

HTTP 응답 헤더에 CSP를 설정하여 허용된 소스에서만 스크립트가 실행되도록 제한한다.

Next.js의 `next.config.js`에서 다음과 같이 설정한다.

| CSP 지시자 | 설정 값 |
|------------|---------|
| default-src | 'self' |
| script-src | 'self' 'nonce-{random}' |
| style-src | 'self' 'unsafe-inline' |
| img-src | 'self' data: https: |

## 5. SSR (Server-Side Rendering) 데이터 보호

### 5.1 Server Component vs Client Component

Next.js App Router에서는 Server Component와 Client Component를 구분하여 민감한 데이터를 안전하게 처리할 수 있다.

| 항목 | Server Component | Client Component |
|------|------------------|------------------|
| 실행 환경 | 서버 | 브라우저 |
| API 직접 호출 | 가능 | 불가능 |
| 민감 정보 접근 | 가능 | 최소화 |
| 환경 변수 | 모든 변수 접근 가능 | NEXT_PUBLIC_* 만 가능 |

### 5.2 민감 데이터 처리 원칙

#### 5.2.1 Server Component에서만 처리

다음 데이터는 반드시 Server Component에서만 처리하고 Client Component로 전달하지 않는다.

- API Secret Keys
- 데이터베이스 연결 정보
- 사용자 개인정보 전체 (이름, 주소, 전화번호 등)
- 결제 정보

#### 5.2.2 데이터 최소화 원칙

Client Component로 전달할 때는 필요한 최소한의 정보만 전달한다.

예를 들어, 사용자 정보를 표시할 때 전체 사용자 객체가 아닌 표시에 필요한 이름과 이메일만 전달한다.

#### 5.2.3 환경 변수 관리

- **서버 전용 환경 변수**: `DATABASE_URL`, `JWT_SECRET`, `API_SECRET_KEY`
- **클라이언트 노출 가능**: `NEXT_PUBLIC_API_URL`, `NEXT_PUBLIC_SITE_NAME`

### 5.3 API Route 보호

#### 5.3.1 인증 미들웨어

모든 API Route는 `middleware.ts`에서 인증을 검증한다. 인증되지 않은 요청은 즉시 401 Unauthorized를 반환한다.

#### 5.3.2 Rate Limiting

Nginx 또는 Edge 레벨에서 Rate Limiting을 적용하여 동일 IP에서 과도한 요청을 차단한다.

- **일반 API**: 분당 60회
- **티켓 구매 API**: 분당 10회
- **대기열 진입**: 초당 100회

## 6. 대기열 토큰 보안

### 6.1 토큰 생성 및 검증

#### 6.1.1 토큰 구조

대기열 토큰은 JWT 형태로 생성되며 다음 정보를 포함한다.

- `queueId`: 대기열 고유 식별자
- `userId`: 사용자 ID
- `position`: 대기 순번
- `eventId`: 이벤트 ID
- `expiresAt`: 토큰 만료 시간
- `signature`: 위변조 방지 서명

#### 6.1.2 토큰 검증 프로세스

- **서명 검증**: 토큰이 서버에서 발급된 것인지 확인
- **만료 시간 확인**: 현재 시간과 비교
- **사용 여부 확인**: Redis에서 이미 사용된 토큰인지 검증
- **권한 확인**: userId와 현재 로그인 사용자 일치 여부

### 6.2 재사용 방지

대기열 토큰은 1회성으로 사용되며, 사용 후에는 Redis에 블랙리스트로 등록된다. 동일한 토큰으로 재접근 시도 시 즉시 차단된다.

### 6.3 중복 탭 접근 방지

사용자가 여러 탭에서 동시에 구매를 시도하는 것을 방지하기 위해, 활성 세션은 Redis에 저장되며 중복 접근 시 경고 메시지를 표시한다.

## 7. 통신 보안

### 7.1 HTTPS 강제

모든 통신은 HTTPS를 통해 암호화되며, HTTP 요청은 자동으로 HTTPS로 리다이렉트된다. Nginx에서 HSTS(HTTP Strict Transport Security) 헤더를 설정하여 브라우저가 항상 HTTPS로 접속하도록 강제한다.

### 7.2 API 요청 암호화

민감한 데이터를 포함한 API 요청은 HTTPS를 통해 전송되며, Request Body는 JSON 형태로 직렬화된다.

- **결제 정보**: PG사 제공 암호화 라이브러리 사용
- **개인정보**: AES-256 암호화 적용 (필요 시)

## 8. 로깅 및 모니터링

### 8.1 보안 이벤트 로깅

다음 이벤트는 반드시 로그로 기록되어야 한다.

- 로그인 실패 (IP, 시간, 시도 횟수)
- 비정상적인 API 요청 (Rate Limit 초과, 잘못된 토큰)
- 대기열 토큰 위조 시도
- CSRF 토큰 불일치

### 8.2 민감 정보 제외

로그에는 비밀번호, JWT 토큰 전체, 결제 정보 등 민감한 데이터를 절대 기록하지 않는다. 사용자 식별이 필요한 경우 `userId`만 기록한다.

## 10. 결론

본 문서에서 정의한 보안 전략을 철저히 준수하여 사용자 데이터를 보호하고 안전한 서비스를 제공한다. 특히 Next.js App Router 환경에서 Server Component와 Client Component를 명확히 구분하여 민감 데이터 유출을 방지하고, HttpOnly Cookie와 CSRF 토큰을 통해 세션 하이재킹 및 CSRF 공격을 차단한다.

또한 XSS 공격 방어를 위해 사용자 입력 검증과 Content Security Policy를 적용하며, 대기열 토큰의 위조 및 재사용을 방지하는 메커니즘을 구현한다. 이러한 다층적 보안 전략을 통해 대규모 트래픽 환경에서도 안정적이고 신뢰할 수 있는 티켓 구매 서비스를 운영할 수 있다.
