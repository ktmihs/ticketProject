# API 계약서 (API Contract Specification)

## 문서 개요

본 문서는 티켓 구매 서비스의 서버 API 명세를 정의한다. 모든 API는 REST 원칙을 따르며, 상태 머신에 정의된 흐름에 따라 호출되어야 한다.

---

## 1. 공통 사항

### 1.1 Base URL

```
Production: https://api.ticket-service.com
Development: http://localhost:3000/api

```

### 1.2 인증 방식

- **Cookie 기반 인증** (HttpOnly, Secure, SameSite=Strict)
- **대기열 토큰**: `X-Queue-Token` 헤더 (Edge에서 검증 후 전달)
- **HOLD 토큰**: JWT 형식 (요청 본문에 포함)

### 1.3 공통 헤더

**요청 헤더:**

```
Content-Type: application/json
X-Queue-Token: <queue_token>  // 구매 API만 필수
Cookie: auth_token=<jwt>       // 인증 필요 시

```

**응답 헤더:**

```
Content-Type: application/json
X-Request-ID: <uuid>           // 요청 추적용

```

### 1.4 공통 에러 응답 형식

모든 에러 응답은 다음 형식을 따른다:

```tsx
interface ErrorResponse {
  error: {
    code: string;           // 에러 코드 (대문자_스네이크_케이스)
    message: string;        // 사용자용 메시지
    details?: any;          // 추가 정보 (선택)
  };
  timestamp: number;        // Unix timestamp (ms)
  requestId: string;        // X-Request-ID 값
}

```

**예시:**

```json
{
  "error": {
    "code": "SEAT_ALREADY_HOLDING",
    "message": "다른 사용자가 이미 선점한 좌석입니다",
    "details": {
      "seatId": 1001,
      "holdUntil": 1737533100000
    }
  },
  "timestamp": 1737532800000,
  "requestId": "req_a1b2c3d4"
}

```

### 1.5 HTTP 상태 코드 정책

| 상태 코드 | 의미 | 사용 예시 |
| --- | --- | --- |
| 200 | 성공 | 조회, 구매 성공 |
| 201 | 생성 성공 | 대기열 진입 |
| 400 | 잘못된 요청 | 필수 파라미터 누락 |
| 401 | 인증 실패 | **토큰이 없거나 만료/위조됨** |
| 403 | 권한 없음 | **토큰은 유효하나 권한이 없음 (대기열 미통과, 차단 등)** |
| 404 | 리소스 없음 | 존재하지 않는 공연 |
| 409 | 충돌 | 좌석 중복 선택 |
| 410 | 만료됨 | HOLD 시간 만료 |
| 422 | 처리 불가 | 결제 실패 |
| 429 | Rate Limit 초과 | 1초 10회 초과 |
| 500 | 서버 오류 | 내부 오류 |
| 503 | 서비스 불가 | Redis/DB 장애 |

**401 vs 403 구분 기준:**

- **401**: 토큰 자체가 없거나, 만료되었거나, 위조된 경우
- **403**: 토큰은 유효하나, 해당 리소스에 접근할 권한이 없는 경우

## 2. 공연 조회 API

### 2.1 공연 목록 조회

**Endpoint:** `GET /shows`

**설명:** 현재 판매 중인 공연 목록을 조회한다.

**요청:**

```
GET /shows?status=available&limit=20&offset=0

```

**쿼리 파라미터:**

| 파라미터 | 타입 | 필수 | 설명 | 기본값 |
| --- | --- | --- | --- | --- |
| status | string | N | 판매 상태 필터 (available, sold_out, upcoming) | available |
| limit | number | N | 페이지 크기 (1-100) | 20 |
| offset | number | N | 오프셋 | 0 |

**성공 응답: 200 OK**

```json
{
  "data": {
    "shows": [
      {
        "id": "show_123",
        "title": "2026 콘서트",
        "description": "최고의 공연",
        "date": "2026-03-15T19:00:00Z",
        "venue": "올림픽공원",
        "seatType": "reserved",
        "price": {
          "min": 50000,
          "max": 150000
        },
        "availability": {
          "total": 5000,
          "remaining": 1234,
          "soldOut": false
        },
        "saleStartAt": "2026-01-22T10:00:00Z",
        "saleEndAt": "2026-03-15T18:00:00Z",
        "thumbnail": "https://cdn.example.com/show123.jpg"
      }
    ],
    "pagination": {
      "total": 15,
      "limit": 20,
      "offset": 0,
      "hasNext": false
    }
  },
  "timestamp": 1737532800000
}

```

**에러 응답:**

**400 - 잘못된 쿼리 파라미터:**

```json
{
  "error": {
    "code": "INVALID_QUERY_PARAMS",
    "message": "limit은 1-100 사이의 숫자여야 합니다",
    "details": {
      "field": "limit",
      "value": 150
    }
  },
  "timestamp": 1737532800000,
  "requestId": "req_a1b2c3d4"
}

```

**500 - 서버 내부 오류:**

```json
{
  "error": {
    "code": "INTERNAL_SERVER_ERROR",
    "message": "서버 오류가 발생했습니다. 잠시 후 다시 시도해주세요"
  },
  "timestamp": 1737532800000,
  "requestId": "req_a1b2c3d4"
}

```

### 3.3 대기열 상태 스트림 (SSE)

**Endpoint:** `GET /queue/stream/:showId`

**설명:** Server-Sent Events로 대기열 상태를 실시간 수신한다.

**⚠️ SSE 지원 불가 환경:** SSE를 지원하지 않는 환경(일부 구형 브라우저, 방화벽 등)에서는 `GET /queue/status` Polling 방식을 사용한다.

**요청:**

```
GET /queue/stream/show_123
Header: X-Queue-Token: <queue_token>

```

**SSE 이벤트:**

**1) position_update (순번 갱신)**

```
event: position_update
data: {"position": 1200, "estimatedWaitTime": 400}

```

**2) allowed (구매 허용)**

```
event: allowed
data: {"queueToken": "new_jwt...", "allowedUntil": 1737533400000}

```

**3) blocked (차단)**

```
event: blocked
data: {"reason": "매크로 의심 행위 감지"}

```

**4) error (연결 오류)**

```
event: error
data: {"message": "서버 연결 실패"}

```

**에러 응답:**

**401 - 토큰 없음/만료:**

```json
{
  "error": {
    "code": "INVALID_QUEUE_TOKEN",
    "message": "유효하지 않은 대기열 토큰입니다"
  },
  "timestamp": 1737532800000,
  "requestId": "req_a1b2c3d4"
}

```

**403 - 차단된 사용자:**

```json
{
  "error": {
    "code": "QUEUE_BLOCKED",
    "message": "차단된 사용자입니다"
  },
  "timestamp": 1737532800000,
  "requestId": "req_a1b2c3d4"
}

```

## 7. 상태 코드별 클라이언트 처리 가이드

| HTTP | 에러 코드 | UI 액션 | Redux 액션 |
| --- | --- | --- | --- |
| 401 | INVALID_QUEUE_TOKEN | 대기열 재진입 안내 | `resetQueue()` → `joinQueue()` |
| 401 | UNAUTHORIZED | 로그인 페이지 이동 | `logout()` |
| 403 | QUEUE_NOT_ALLOWED | "대기 후 이용 가능" 모달 | `resetPurchase()` |
| 403 | QUEUE_BLOCKED | "차단됨" 페이지 (고객센터) | `blockUser()` (영구) |
| 409 | SEAT_ALREADY_HOLDING | "다른 좌석 선택" 토스트 | `holdSeatFailure()` |
| 409 | ALREADY_PURCHASED | "구매 완료" 페이지 이동 | `resetPurchase()` |
| 410 | HOLD_EXPIRED | "선점 만료" 모달 | `expireHold()` → `returnToSeatSelect()` |
| 410 | QUEUE_EXPIRED | "시간 만료" 모달 | `expireToken()` |
| 422 | PAYMENT_FAILED | "결제 실패" 재시도 버튼 | `failPayment()` |
| 429 | RATE_LIMIT_EXCEEDED | "잠시 후 재시도" 토스트 | N/A (1초 대기) |
| 503 | SERVICE_UNAVAILABLE | "일시적 오류" 30초 후 재시도 | `joinFailure({ reason: 'SERVER_ERROR' })` |

## 11. 포트폴리오 강조 포인트

본 API 계약서는 다음 설계 원칙을 반영한다:

1. **명확한 에러 구분**: 401(인증 실패) vs 403(권한 없음) 기준을 일관되게 적용
2. **서버 기준 검증**: HOLD 만료는 클라이언트 타이머가 아닌 서버 시각 기준으로 최종 판정
3. **상태 머신 기반**: Queue/Purchase 상태 전이와 정확히 매핑되는 에러 코드 설계
4. **Redis/DB 역할 분리**: 좌석 미지정(Redis), 좌석 지정(DB) 각각의 장애 시나리오 명시
