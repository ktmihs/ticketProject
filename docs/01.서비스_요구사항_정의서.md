# 서비스 요구사항 정의서 (SRD)

## 1. 문서 개요

### 1.1 목적

본 문서는 **대규모 트래픽 환경을 가정한 티켓 구매 웹 서비스**의 기능적·비기능적 요구사항을 정의한다. 본 프로젝트는 단순 기능 구현이 아닌, **프론트엔드 중심의 아키텍처 설계 능력**, **상태 관리 전략**, **보안 고려**, **실제 서비스 수준의 흐름 설계**를 목표로 한다.

### 1.2 프로젝트 목표

- 대기열(Queue) 기반 티켓 구매 흐름 설계
- Edge → Nginx → API Server로 이어지는 요청 흐름 이해
- Next.js(App Router) 기반 보안 전략 정리
- RTK / React Query의 역할 분리 및 데이터 관리 기준 확립
- “아키텍처를 설명할 수 있는 프론트엔드 포트폴리오” 산출

### 1.3 기술 스택 요약

| 영역     | 기술                                               |
| -------- | -------------------------------------------------- |
| Frontend | Next.js(App Router), TypeScript                    |
| State    | Redux Toolkit, React Query                         |
| Backend  | Express.js                                         |
| Infra    | CloudFront 또는 Cloudflare Workers, Nginx          |
| Auth     | Cookie 기반 인증 (HttpOnly)                        |
| 기타     | Redis(Queue/Cache, 선택), WebSocket(SSE 대안 가능) |

---

## 2. 서비스 개요

### 2.1 서비스 설명

사용자는 공연/이벤트의 티켓을 구매할 수 있다. 특정 시점에 대량의 사용자가 몰리는 상황을 가정하여 **대기열 시스템**을 적용한다.

구매 흐름은 다음과 같다.

> 사용자 → Next.js → Edge → Nginx → Express API

### 2.2 주요 사용자

- **일반 사용자**: 티켓 조회, 대기열 진입, 티켓 구매
- **운영자(Admin)** (확장): 이벤트 및 좌석 관리

---

## 3. 기능 요구사항

### 3.1 사용자 공통

#### 3.1.1 이벤트 조회

- 사용자는 현재 판매 중인 이벤트 목록을 조회할 수 있다
- 이벤트 정보

  - 이벤트 ID
  - 제목
  - 공연 일시
  - 판매 시작 시간
  - 잔여 티켓 수

#### 3.1.2 이벤트 상세 조회

- 사용자는 특정 이벤트의 상세 정보를 확인할 수 있다
- 좌석 타입별 가격 및 잔여 수량 노출

---

### 3.2 대기열 시스템

#### 3.2.1 대기열 진입

- 판매 시작 시간 이전 접근 시 대기열 페이지로 이동
- 판매 시작 시점에 Edge 또는 서버에서 대기열 토큰 발급

#### 3.2.2 대기 상태 유지

- 대기 번호 및 예상 대기 시간 표시
- 주기적인 상태 갱신 (Polling / SSE)

#### 3.2.3 구매 가능 상태 전환

- 순서가 도래하면 구매 페이지 접근 허용
- 제한 시간 내 구매하지 않으면 대기열 재진입

---

### 3.3 티켓 구매

#### 3.3.1 구매 페이지 접근 제어

- 대기열 토큰 검증 성공 시 접근 가능
- 토큰은 서버에서만 검증 가능

#### 3.3.2 좌석 선택

- 좌석 타입 선택 (선착순)
- 재고 부족 시 즉시 실패 처리

#### 3.3.3 결제 요청 (Mock)

- 결제 성공/실패 시나리오 분기 처리
- 결제 완료 후 구매 완료 페이지 이동

---

### 3.4 사용자 세션 관리

- 새로고침 시 대기열 상태 유지
- 중복 탭 접근 시 구매 제한

---

## 4. 비기능 요구사항

### 4.1 성능

- 동시 접속자 수 증가 시 서버 부하 최소화
- 정적 리소스는 CDN에서 제공
- API 요청은 Nginx를 통해 제어

### 4.2 확장성

- 대기열 로직은 서버/Edge 단 분리 가능해야 함
- Redis 등 외부 스토리지로 확장 가능

### 4.3 안정성

- 대기열 토큰 위조 방지
- API Rate Limit 적용

---

## 5. 데이터 관리 전략

### 5.1 프론트엔드 상태 분류 기준

| 구분         | 관리 위치   | 예시                     |
| ------------ | ----------- | ------------------------ |
| UI 상태      | 지역 state  | 모달, 탭, 선택 상태      |
| 전역 UI 상태 | RTK         | 로그인 상태, 대기열 상태 |
| 서버 데이터  | React Query | 이벤트 목록, 좌석 정보   |
| 캐시 데이터  | React Query | 이벤트 상세              |

### 5.2 Redux Toolkit 사용 대상

- 인증 상태
- 대기열 상태 (position, status)
- 전역 UI 제어

### 5.3 React Query 사용 대상

- 이벤트 목록 조회
- 좌석 잔여 수량 조회
- 구매 가능 여부 확인

---

## 6. 보안 요구사항 (Next.js 기준)

### 6.1 인증/인가

- 인증 정보는 **HttpOnly Cookie** 사용
- localStorage/sessionStorage 사용 금지

### 6.2 서버 컴포넌트 활용

- 민감 데이터는 Server Component에서만 접근
- Client Component에는 최소 정보만 전달

### 6.3 API 보호

- Middleware에서 인증 검사
- 구매 API는 서버에서만 접근 가능

### 6.4 Edge 보안

- Edge에서 트래픽 필터링
- 비정상 요청 사전 차단

---

## 7. 아키텍처 흐름 요약

1. 사용자 요청 → Next.js
2. 정적 리소스 → CDN
3. 대기열 요청 → Edge
4. API 요청 → Nginx → Express
5. 상태 관리 → RTK / React Query
